# Refactoring Hand-Over

## Hand-Over From Previous Phase

The `refactor-split-to-pkg-and-cmd` branch refactors `go-gisty` while preserving
the existing public `gisty` package API.

The public package remains a compatibility facade. Dependency-specific and
pure logic responsibilities now live in focused internal packages:

- `internal/ghcmd`: shared external `gh` execution and in-process Cobra command
  setup for both the library and CLI.
- `gisty/internal/httpclient`: GitHub CLI-compatible HTTP client construction
  and token lookup.
- `gisty/internal/gistid`: gist selector sanitization.
- `gisty/internal/gistinfo`: parsing for tab-separated `gh gist list` output.

The branch also adds `cmd/gisty`. It forwards:

```console
gisty <args...>
```

as:

```console
gh gist <args...>
```

The Maintainer selected transparent-wrapper semantics for the CLI. When the
child `gh` process returns an exit code, `gisty` preserves it. If `gh` cannot be
started and no child exit code exists, `gisty` returns status `1`. This behavior
is documented in `README.md` and covered by hermetic subprocess tests.

Backward compatibility remains intentionally prioritized. The public
`AltFunc` type still exposes GitHub CLI option types because changing that API
would require a future major version.

## Deferred Follow-Ups

- Consider a future major-version API that hides GitHub CLI option types from
  `AltFunc`. Exposing upstream option types preserves backward compatibility
  today but limits future refactoring freedom.
- Consider replacing the `repo sync`-based gist update path if GitHub CLI adds
  a dedicated gist edit workflow suitable for this library.
- Consider adding an optional CI-level process-boundary contract check on a
  controlled platform. Keep it separate from unit tests. A temporary `gh` stub
  on `PATH` could verify runtime argv, stderr, and exit-code forwarding without
  requiring GitHub authentication.
